# Анализатор логов (log_analyzer.py / `sheptun analyze-replacements`)

Грабли и решения по LLM-пайплайну через прокси (base_url из `SHEPTUN_ANTHROPIC_BASE_URL`).

## Прокси ломает SSE-стрим: разбираем поток сами
Прокси шлёт content-блоки с непоследовательными индексами (0, потом 2) — штатный
SDK-аккумулятор (`.stream()`/`get_final_message()`) индексирует снапшот по `event.index`
и падает с `IndexError: list index out of range` на КАЖДОМ запросе. Поэтому `_ask_streaming`
разбирает сырой поток событий сам (index-агностично, склеивает `text_delta`), не используя
аккумулятор SDK. Не возвращать к `.stream()`.
Связано: `src/sheptun/log_analyzer.py` (`_ask_streaming`, `_consume_events`, `_StreamResult`).

## Пустой out=0/stop=max_tokens = залипание прокси → ретрай
Изредка (~0.5%) прокси под нагрузкой залипает на минуту и возвращает пустой ответ с
`stop=max_tokens`. Раньше это считалось «успешным пустым батчем» и чекпоинт проезжал мимо.
Теперь `_guard_truncated` бросает исключение при пустом тексте + `stop==max_tokens`
(константа `_MAX_TOKENS_STOP`), и срабатывает существующий ретрай с backoff — обычно
следующая попытка проходит. Непустой усечённый ответ НЕ бросаем (парсим сколько есть).
`max_tokens` в коде = 12000 (в `.env` не переопределён), т.е. проблема не в лимите —
крутить `CONCURRENCY`/`BATCH_SIZE`, а не `max_tokens`.
Связано: `src/sheptun/log_analyzer.py` (`_guard_truncated`, `_suggest_with_retry`).

## Ответ без внешних [ ] / усечённый: fallback-разбор объектов
Модель РЕГУЛЯРНО шлёт JSON без обёртки `[ ]` (объекты через запятую) или обрезанный по
токенам. Цельный `json.loads` это отвергает → `_extract_items` раньше возвращал 0 и весь
батч терялся — это и есть причина «несколько итераций без единого совпадения», хотя модель
кандидатов НАХОДИТ. Проблема НЕ в промпте. Теперь есть fallback `_scrape_objects`:
регексом `\{[^{}]*\}` выскребает каждый целый `{...}` отдельно, битый хвост отбрасывается.
Связано: `src/sheptun/log_analyzer.py` (`_extract_items`, `_scrape_objects`).

## Отладочное логирование в analyzer.log
`SHEPTUN_ANALYZER_LOG_PROMPTS=true` пишет в `analyzer.log` (не в `sheptun.log`, чтобы не
засорять вход анализатора) полную цепочку батча на DEBUG: `Промпт (user)` → `Сырой ответ
(N симв, stop=…)` → причины дропов кандидатов → сводку `Кандидатов: X, принято: Y,
отброшено: Z`. Флаг САМ поднимает уровень логгера `sheptun.analyzer` до DEBUG (иначе
уровень INFO глушит debug). `system`-промпт не логируется (статичен). WARNING «Ответ не распарсился
ни в одно правило» — ВСЕГДА, без флага. Финальный дедуп/confidence-отсев (`_accept_new` в
`ReplacementAnalyzer`) в лог НЕ пишется — видно вход и выход `suggest`, но не отсев между
батчами.
Связано: `src/sheptun/log_analyzer.py`, `src/sheptun/cli.py:_setup_analyzer_logging`.

## Тесты анализатора не гонять на живом replacements.yaml
Прогоны с `--apply` на реальном `src/sheptun/config/replacements.yaml` дописывают в него
правила → появляется «фантомное» изменение в `git status`, которое легко случайно
застейджить. НЕ коммитить это (пользователь просил), перед коммитом `git checkout` файла;
для проверок гонять на копии во временной папке.
Связано: `src/sheptun/config/replacements.yaml`.
