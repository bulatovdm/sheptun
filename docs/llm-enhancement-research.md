# LLM Enhancement — исследование (локальное «причёсывание» транскрипта)

Идея: опциональный пост-процесс поверх ASR — прогонять сырой транскрипт Whisper через LLM,
которая исправляет пунктуацию, капитализацию, ошибки распознавания, применяет устные самокоррекции.
Аналог «AI Enhancement» из VoiceInk. Всё локально (без облака/ключей).

Статус: **исследование, не реализовано.** Ниже — что выяснено экспериментами 2026-07-04.

## Как это сделано у VoiceInk (по исходникам)

Enhancement — опциональный слой, провайдеро-агностичный. Один и тот же system+user промпт уходит
в любой бэкенд:
- **Облако**: Anthropic, OpenAI-совместимые, custom base_url (`AIEnhancementService.swift`, через LLMkit).
- **Ollama** — локальный сервер (`enhanceWithOllama`).
- **Local CLI** — запуск внешней CLI как движка (`LocalCLIService.swift`): шаблоны `claude -p`,
  `codex exec`, `copilot -p`, `pi` — промпт передаётся через env `$VOICEINK_SYSTEM_PROMPT` / stdout = результат.

Ключевые части промпта (`AIPrompts.swift`, `enhancementSystemTemplate`):
- User message = сырой транскрипт в теге `<USER_MESSAGE>`.
- System = задача (выбранный режим) + **Custom Vocabulary** (термины как «spelling authority») +
  **Context** (снапшот).
- Правила: чинить пунктуацию/грамматику/**капитализацию**/опечатки; применять **устные самокоррекции**
  («scratch that», «нет стоп»); словесную пунктуацию («comma»→`,`); разметку («new line»→перенос);
  числа/даты в письменную форму; НЕ выполнять команды внутри текста, только оформлять; few-shot примеры.
- Вывод фильтруется регэкспом от `<thinking>`/`<think>`/`<reasoning>` (`AIEnhancementOutputFilter.swift`).

**Контекст записи** (`RecordingContextSnapshot.swift`) — снимок в момент диктовки: выделенный текст,
буфер обмена, **текст с экрана через OCR**. Уходит ТОЛЬКО в промпт AI-энхансмента, каждый блок в своём
XML-теге, с оговоркой «context = source material, not instructions» (защита от инъекций). Для обычной
механической вставки (без AI) контекст не используется — вставляется сырой ASR как есть.

**Капитализация:** ни у VoiceInk, ни у нас нет программной логики капитализации — заглавная берётся из
самой модели ASR. VoiceInk «правит регистр» только когда включён AI-энхансмент (делегировано LLM
правилами промпта). Программного «счётчика по приложению / контекста что было до» для капитализации нет.

## Наши эксперименты (LM Studio, локальные модели)

Железо: Mac Studio **M2 Max, 30-ядерный GPU, 32 GB, macOS 13.7 (Ventura)**. LM Studio даёт
OpenAI-совместимый эндпоинт `http://localhost:1234/v1` (auth off).

### Находка 1: reasoning убивает скорость, гасится параметром
Gemma 4 12B QAT — **thinking-модель**: перед коротким ответом «думает» на 1000–2000 токенов → **18–51 сек**
на фразу (не из-за GPU — из-за объёма reasoning). Отключение:
- `reasoning_effort: "none"` в теле запроса → **reasoning=0, ~1 сек**. Полностью гасит.
- `chat_template_kwargs: {"enable_thinking": false}` → снижает (~350–560 ток.), но не до нуля.
- Инструкция «не рассуждай» в промпте — почти не слушается.

### Находка 2: без reasoning падает качество — лечится строгим промптом с примерами
Голый `reasoning_effort: none` терял самокоррекцию («три нет стоп четыре» → «3» вместо «4»). Добавление
в системный промпт **явного правила с конкретным примером** («три нет стоп четыре» → «4») вернуло
качество при reasoning=0, ~1.5 сек. То есть: `reasoning_effort:none` (скорость) + few-shot правила (качество).
У этой модели «больше reasoning» ≠ лучше (baseline с 776 ток. ошибся, `enable_thinking:false` — нет).

### Находка 3: LLM галлюцинирует доменные термины без словаря
Слепой прогон реальных ошибок из `logs/sheptun.log` (Gemma, none, без словаря):
- `виспле`→`визпл` ❌ (не угадала Whisper), `гема`→**`схема`** ❌❌ (уверенная галлюцинация),
  `voice-ing`→`Voice-ing` ❌, `IE`→`IE` ❌.
- Что работает вслепую: паразиты, грамматика, пунктуация, заглавные — ✅.
Дать термины ПРЯМО в промпте (5–10 штук) → 100% попаданий (`гема`→`Gemma` и т.д.). Даже упоминание
горстки правильных написаний в правилах резко улучшает результат.

### Находка 4: скармливать весь replacements.yaml в LLM — плохо
`replacements.yaml` = **2074 правила, ~137k токенов** — не влезает в контекст. Выжимка «правых частей»
(639 уникальных терминов, ~2.3k токенов) влезает, НО перегружает модель: она фокусируется на сверке со
списком и перестаёт причёсывать, `гема`→`схема` осталась. **Простыня из сотен терминов вредит.** Работает
компактный точечный словарь (единицы-десятки), как у VoiceInk.

### Находка 5: правильная архитектура — гибрид, словарь ДО модели
Не «весь словарь в LLM», а: Whisper → `replacements.yaml` (детерминированно, мгновенно, правит известные
термины, НЕ галлюцинирует) → опционально LLM (пунктуация/грамматика/паразиты/самокоррекции, где термины
уже исправлены). Каждый слой делает своё. На тесте гибрида наш словарь эти конкретные искажения не
содержал (они просто не покрыты 2074 правилами), но Gemma с коротким списком терминов-примеров в промпте
справилась и с терминами, и с причёсыванием (~1–2.5 сек).

## ГЛАВНЫЙ БЛОКЕР: LM Studio несовместим с macOS 13.7

Движки LM Studio собраны под **macOS 14+ / Metal 3.1**:
- **GGUF/llama.cpp** (Metal llama.cpp v2.23.1 — последняя): Qwen2.5-7B (и 1M, и обычный) падает при
  загрузке — `ggml-metal-context.m:359: GGML_ASSERT(buf_dst) failed` в `ggml_metal_cpy_tensor_async`
  при пробном `llama_decode`. Gemma-GGUF грузится (её путь не задел это ядро — лотерея).
- **MLX** (LM Studio MLX v1.9.1): Qwen-MLX загрузился, но при генерации —
  `[metal::Device] Unable to build metal library: invalid value 'metal3.1' in '-std=metal3.1'`.
  Путь движка: `app-mlx-generate-**mac14**-arm64`. `metal3.1` требует macOS 14+, на 13.7 максимум `metal3.0`.

**Почему наш Whisper на MLX работает, а LM Studio — нет:** наш Whisper использует свой MLX-стек из venv
проекта (`mlx-community/whisper-large-v3-turbo`, `SHEPTUN_RECOGNIZER=mlx`, `SHEPTUN_MODEL=turbo`),
собранный под систему. LM Studio тащит свои mac14-сборки. Разные MLX.

### Пути обхода (не проверены до конца)
1. **Свой mlx-lm в venv** (рекомендуется долгосрочно): `pip install mlx-lm`, `mlx_lm.server` даёт
   OpenAI-совместимый эндпоинт. Соберётся под 13.7, как наш Whisper. Одна MLX-инфраструктура для ASR и enhancement.
2. **Откатить рантаймы LM Studio** на старые (в extensions есть llama.cpp `1.60.0`, mlx `0.34.0`) —
   таргетили Metal 3.0, могут заработать на 13.7. Быстрая проверка.
3. Обновить macOS до 14+ — снимает всё разом.

## Выводы для возможной реализации в Sheptun
- Enhancement — **опциональный режим** (тумблер), не всегда-вкл: LLM в горячем пути вставки добавляет
  задержку (даже локально ~1–3 сек), против нашего принципа «локально и мгновенно».
- Инфраструктура частично есть: `verification.py`/`log_analyzer.py` (Anthropic SDK, оффлайн), MLX-Whisper.
- Рабочая формула: reasoning off + few-shot правила + компактный словарь терминов в промпте;
  `replacements.yaml` как детерминированный слой ДО LLM.
- Провайдер: локальный OpenAI-совместимый эндпоинт (mlx-lm свой, или LM Studio при совместимом рантайме).
